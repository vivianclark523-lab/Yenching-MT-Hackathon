# Skill 2 PRD · 智能外卖与采购助手

## 1. 产品定位

**Skill 名称**：`meal-grocery-assistant`

**一句话定义**：在饭点、明确品类、选择困难和周期性采购场景下，基于用户历史订单、天气、日程、优惠和本地高分供给，给出可直接确认的一键下单方案。

Skill 2 的核心不是“再做一个搜索框”，而是把用户每天反复消耗的决策成本压缩成三种产品形态：

1. **极度繁忙：直接代订候选**  
   给一个最稳方案，优先来自用户常点、可信、不踩雷、不与昨日重复，并结合天气和日程避开明显不合适品类。

2. **纠结选择：灵感卡片**  
   用左右滑动推荐卡片承载 4 张方案：高性价比/品质优选 × 稳妥快餐/探索小吃。每张卡必须说明推荐理由和价格预期。

3. **明确品类：常用 + 探索双方案**  
   用户已经说“想吃火锅/麻辣烫/轻食”时，不再让用户筛选列表，而是直接给“最常用方案”和“口碑探索方案”。

Skill 同时覆盖小象超市等采购补货：水、纸、猫粮、驱蚊液等周期性消耗品，按照历史购买周期和季节触发补货建议。

## 1.1 用户路由方式

用户不需要主动声明自己属于哪一种情况。系统根据自然语言自动分流，并在推荐末尾提供快捷路由，帮助用户一键改变方向。

**路由优先级**：

1. 明确采购：猫粮、纸、水、补货 → 采购补货
2. 明确品类：火锅、麻辣烫、面、轻食 → 明确品类
3. 明确代订：很忙、直接帮我下单、别问了 → 极度繁忙
4. 灵感纠结：今天吃什么、不知道吃什么、再挑一挑 → 纠结场景
5. 模糊外卖："帮我点外卖" → 默认极度繁忙，给一个稳妥方案

**推荐末尾快捷路由**：

```text
快捷选项：
1. 我很忙，直接帮我下单外卖
2. 我不满意，还想再挑一挑
3. 我想吃某一品类，你帮我推荐一下
```

这三个快捷入口分别对应：`busy`、`inspiration`、`category`。其中第三个需要用户补充品类后再继续推荐。

## 2. 用户问题

- 饭点时没精力挑外卖，希望“别出错，帮我定”。
- 选择困难时既想稳，又想试点新东西。
- 有明确品类目标，但懒得在评分、配送、价格、优惠里反复筛。
- 常点店有券或促销时，用户不会主动发现。
- 日用品经常等到用完才想起来买。
- 旅游状态下，用户希望吃到当地特色，而不是被推荐平时常吃的连锁快餐。

## 3. 输入数据与实现方式

### 3.1 数据来源

- `mocks/restaurants.json`：餐厅基础信息、评分、人均、营业时间、位置、品类。
- `mocks/coupons.json`：限时券和满减券，按虚拟时间判断是否可用。
- `mocks/meal_grocery.json`：Skill 2 专用的历史外卖、菜品、配送、用户偏好、采购周期、旅行特色标签。
- `mocks/clock.py`：唯一虚拟时钟入口，保证沙盒拨时间后推荐结果可复现。

### 3.2 执行脚本

脚本入口：

```bash
python3 skills/meal-grocery-assistant/scripts/meal_context.py recommend --scenario busy
python3 skills/meal-grocery-assistant/scripts/meal_context.py recommend --scenario inspiration
python3 skills/meal-grocery-assistant/scripts/meal_context.py recommend --scenario category --category "火锅"
python3 skills/meal-grocery-assistant/scripts/meal_context.py grocery --need auto
python3 skills/meal-grocery-assistant/scripts/meal_context.py profile
```

脚本只输出 JSON，由 LLM 按 `SKILL.md` 的标准模板转成用户可读话术；脚本不调用真实支付、不自动提交订单。

### 3.3 OpenCloud 适配

- 纯 Python 3.10+ 标准库，无额外依赖。
- 无 macOS API、无本机绝对路径。
- 所有时间读取走 `mocks.clock.virtual_now()`。
- 缺少真实外部 API 时仍可通过 mock 数据跑完整 demo。

## 4. 三个核心场景

### 4.1 极度繁忙场景：无暇自选，直接代订

**目标**：给用户一个可以直接确认的稳妥方案。

**核心逻辑**：

- 优先选择用户历史常点店和常点菜。
- 排除昨日已点店铺或同款主菜。
- 天气炎热时避开火锅、羊蝎子等重热品类，优先凉面、酸菜鱼、轻快餐、饮品搭配。
- 如果日程显示明天有火锅局，今天不再推荐火锅。
- 保留高评分、配送稳定、预制菜风险低、差评风险低的店。

**产品形态**：

```text
今晚我直接给你配一个稳的：
<店名>｜<菜品>｜预计 ¥<价格>
理由：你常点、评分 <rating>、今天不和昨天重复；<天气/日程避坑理由>

要我按这个下单吗？
```

### 4.2 纠结场景：需要灵感，辅助决策

**目标**：让用户用轻交互快速做选择。

**展示形态**：四张左右滑动卡片。

**卡片矩阵**：

| 价格档 | 稳妥方案 | 探索方案 |
|---|---|---|
| 高性价比 | 常吃/快餐/低风险 | 当地火爆、高分、未尝试小吃 |
| 品质优选 | 高评分正餐/连锁稳定 | 高评分新店/特色菜 |

**每张卡必须包含**：

- 店名、菜品、预计实付、评分。
- 推荐理由，必须引用具体数据：历史常点、评分、券、天气适配、当地特色、未尝试。
- 用户画像标签：是否偏好连锁、是否反感预制菜、口味偏好、堂食/外卖倾向。
- 旅游状态下，至少一张卡必须是当地特色方案。

### 4.3 明确品类场景：已有目标，懒得筛选

**目标**：用户说“我想吃火锅，帮我点一下”时，直接给两个可选方案。

**方案一：常用方案**

- 从历史数据中找用户最常点的该品类或近似品类店铺。
- 如果该店今日有券，优先展示。
- 如果昨日已点或日程冲突，降级为备选并说明原因。

**方案二：探索方案**

- 找同品类高评分、用户未尝试、与画像匹配的店。
- 过滤低评分、配送慢、明显不符合天气/日程的选项。

**产品形态**：

```text
想吃 <品类> 的话，我给你两个：

常用：<店名>｜<菜品>｜¥<价格>
理由：你点过 <N> 次，稳定不踩雷

探索：<店名>｜<菜品>｜¥<价格>
理由：<rating> 分，你没试过，口碑更好/更符合今天状态

选哪个？我再给你确认订单。
```

## 5. 安全与确认

- 所有真实消费动作前必须二次确认。
- 缺少地址、人数、忌口、预算时，能默认就默认；影响安全或会明显踩雷时才追问。
- 不展示 JSON、脚本路径、字段名、技术细节。
- 不收集真实隐私；demo 只使用 mock 或用户显式输入。
- 推荐理由不能写“感觉不错”，必须来自数据或用户画像。

## 6. 验收标准

- `SKILL.md` frontmatter 合规，目录名和 `name` 一致。
- `meal_context.py` 可命令行执行，所有输出为 JSON。
- 三个核心场景都能跑出结果。
- 热天避火锅、昨日去重、明日日程避重、旅游状态含当地特色均可被脚本体现。
- 采购补货能根据 mock 周期给出补货建议。
