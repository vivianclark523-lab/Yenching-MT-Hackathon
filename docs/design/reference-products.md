### 1.2 它解决的 3 个用户场景（README 原文）

1. **定时取号**：让 agent 在指定时间自动打开取号页，并按页面流程取号
2. **定时查看已排桌数，达到条件再取号**：让 agent 按用户指定的频率查看当前排队状态；未达阈值时不打扰，达到后自动进入取号流程
3. **取号成功后持续刷新桌数，到点提醒出发**：取号后按频率刷前方等待桌数；到设定值时提醒用户出发

### 1.3 SKILL.md 结构（极简，与 MT-Paotui 风格差异显著）

完整 SKILL.md 只有 ~160 行，结构：


**对比 MT-Paotui**：
- ❌ 没有 `⚠️ 输出规范` callout block
- ❌ 没有 `⚠️ 强制重读`
- ❌ 没有显式风险提示和免责声明 section（合规声明在 README 而不是 SKILL.md）
- ✅ 没有 references/ 子目录（160 行直接放在 SKILL.md 里）
- ✅ 信息密度更高、更适合"小而精"型 Skill

### 1.4 8 个标准回复模板（直接引用，可借鉴）

> "**向用户输出时必须使用'标准回复格式'，不要自由发挥，不要输出脚本 JSON、运行日志、字段清单或多余解释。**"

| 场景 | 模板（原文） |
|---|---|
| 取号成功 | `已取号成功：<店名>` `桌号：<tableNumDesc>` `前方等待：<queueWaitTableNum> 桌` `人数：<peopleCount> 人` |
| 只看状态 | `当前排队：<店名>` `状态：<queueStatusText>` `总等待：<queue.totalWaitCount> 桌` `桌型: - <name>（<capacity>）：<waitCount> 桌` |
| 取号后提醒出发 | `可以出发了：<店名>` `前方等待：<queueWaitTableNum> 桌` |
| 多个候选 | `找到多个可能的门店，请回复编号：` `1. <店名>｜<地址>` |
| 短信验证码 | `请输入短信验证码。` |
| 浏览器验证 | `请在打开的浏览器页面完成验证，完成后告诉我继续。` |
| 询问检查频率 | `请告诉我多久查一次排队进度。` |
| 业务失败 | `无法完成：<原因>`（双层文案时 `主文案：副文案`） |

### 1.5 执行边界（直接引用，是 dianping 最有价值的部分）

> "**状态读取脚本只执行一次读取，不常驻轮询。定时唤醒由 agent 所在环境的自动化、定时任务或提醒系统负责。**"

→ 这就是回答我们 docs/openclaw-architecture.md 第 8 节 ❓2 的金句

> "**阈值取号和取号后提醒出发场景中，必须先确认用户希望多久查一次排队进度。**"

→ 不要默认值，让用户决定频率

> "**未达到阈值时静默结束，不要向用户报数。**"

→ 关键 UX：不打扰用户，只在临界点出现

> "**用户未提供人数时默认 `2` 人。**"

→ 极少数允许默认值的情况——"2 人" 在餐厅排队场景是合理猜测

### 1.6 候选和断点（直接引用，处理复杂场景的妙招）

> - "**`candidates` 只有 1 个：直接运行 `confirm-search`。**"
> - "**`candidates` 有多个：向用户展示候选编号，让用户回复编号后再继续。**"
> - "**用户回复编号后，不要复用上一页，不要重新搜索；直接用该编号运行 `confirm-search --index "<编号>"`。**"
> - "**如果返回 `browser_verification_required`，让用户在浏览器页面完成验证；完成后重新运行同一条命令。**"
> - "**如果返回 `verification_required`，让用户提供短信验证码后续跑同一命令并追加 `--sms-code "<SMS_CODE>"`。**"

→ 这是非常聪明的"断点处理"模式：用户输入是 inline 的，不需要复杂的 state machine

### 1.7 本地缓存策略（stores.json 设计）

```json
{
  "stores": [
    {
      "dpShopId": "1000000000",
      "dpShopIdEncrypt": "sample-encrypted-shop-id",
      "storeName": "示例火锅店（测试门店）",
      "aliases": ["示例火锅", "测试门店"],
      "address": "示例城市示例区示例路 100 号",
      "lastVerifiedAt": "2026-01-01T00:00:00Z"
    }
  ]
}
```

**设计要点**：
- 第一次取号后入库
- 后续直接报店名（命中缓存）→ 跳过浏览器搜索
- 有 `aliases` 数组支持多种叫法（"海底捞"/"hdl"/"海底捞王府井店"）
- `lastVerifiedAt` 时间戳支持过期检测

### 1.8 一个真正聪明的用户引导（README 原文）

> "**重要：对于第一次取号的店铺，建议直接发送大众点评分享链接。**"
> "**这样可以跳过浏览器搜索步骤，直接获取店铺 ID，省下不必要的 token 消耗。**"

→ 不要让 AI 解析自然语言店名 → 直接用结构化 URL → 又快又准
→ **我们的 Skill 1 应该照搬这条**

### 1.9 合规边界（README 9 条声明，对我们有启发）

> - 仅面向真实用户的个人自用场景
> - 仅使用用户本人账号的正常登录态
> - 仅使用浏览器在用户授权后获取的真实信息
> - 不支持手动指定、伪造、模拟或覆盖地理位置
> - 不支持绕过店铺距离限制、登录验证、短信验证、验证码或平台风控
> - **不支持批量取号、多账号取号、代取号、抢号、刷号或任何商业化使用**
> - 当页面提示需要到店附近、距离过远、排队关闭、无需排队、商家暂停取号或需要人工验证时，工具会停止自动流程并返回相应提示

→ **公开声明使用边界**是个好实践，hackathon 评委会觉得我们"懂规矩"

### 2.5 工作流程的标准化（Step 0-5，我们的 Skill 都该这样组织）

| Step | 职责 | 关键设计 |
|---|---|---|
| Step 0 | 登录授权检查 | 接口返回 code:10000 → 自动 `login --force` |
| Step 1 | 信息预收集（**静默**） | "**严禁在登录检查前追问用户任何信息**" |
| Step 2 | 拉取地址簿 | 优先地址簿匹配 → 缓存缺失才走 POI 搜索 |
| Step 3 | 地址确认 | 序号 / 关键词 / 新地址三种用户回复路径 |
| Step 4 | 物品参数构造 | 按 2.3 映射表查；模糊描述"一些东西"→ 直接归"其他"，**不追问** |
| Step 5 | **预览 → 确认 → 提交** | **强制两步**：先预览展示费用卡片，用户回"确认"后才加 `--confirm` 提交 |

### 2.7 错误码处理（references/errors.md）

| 错误码 | 现象 | 启发 |
|---|---|---|
| 10000 | Token 服务端过期 | 我们的 Mock 也要支持"凭证过期"场景演示 |
| 10001 / 10113 | 城市未开通 | Mock 数据要包含"该城市暂未开通"剧本 |
| 10112 | 区域未开通 | 不是所有地址都能配送 |

| 10115 | 配送距离过远 | 距离上限概念 |

| 11004-11008 | 风控拒绝 | houseNumber / name / remark 必须留空 |

### 2.8 安全门控（**直接搬到我们的 Skill**）

```
1. 强制两步确认：先预览展示费用卡片，等用户明确回复"确认"后才加 --confirm 提交。
2. 信息完整性：取件地址、收件地址、联系电话缺一不可。
3. 金额异常拦截：费用 > 100 元时需额外向用户确认。
```
