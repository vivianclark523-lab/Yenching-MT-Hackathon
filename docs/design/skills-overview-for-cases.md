## §1 Skill 1 · 多店并行排队管家（`watch-restaurant-queues` 🔔）

**一句话**：后台同时盯多家候选餐厅排队，到点提醒出发，必要时（确认后）自动取号。

**怎么被触发**
- 被动（用户开口）：「今晚吃 X 或 Y」「帮我盯一下排队」「什么时候该出门」「取号了吗」「前面还有几桌」
- 主动（服务找人，4 条同级入口）：
  - A 用户主动表达候选店
  - B 观察到用户在美团 APP 订座 → 静默启动该店监控
  - C 临近惯常饭点（如每周五 17:00 常外出）→ 主动问"要不要先盯排队？"
  - D 用户曾提过的店今晚恰好出排队窗口 → 主动推送"今晚 7 点估计能排到，要盯吗？"

**核心能力（输入 → 产出）**
1. 搜店确认（接受大众点评分享链接，跳过搜索省 token）
2. 并行监控多店实时排队（mock + 虚拟时钟），**未达阈值时静默不打扰**
3. 达阈值 → 取号前两步确认 → 自动取号
4. 持续追踪前方桌数 → 剩 N 桌（默认 4）主动推送出发提醒（含高德真实步行/打车路径 + 打车费）
5. 联动叫车

**安全门控（case 会用到的"卡点"）**：取号前必须确认；缺店名要问；人数默认 2；多店同时达阈值要问"先给哪家取"；监控前必问"多久查一次"。

## §2 Skill 2 · 智能外卖与采购助手（`meal-grocery-assistant` 🍱）

**一句话**：饭点/补货场景的主动决策助手——少问、少筛，直接给可确认的一键下单方案。

**怎么被触发**
- 被动（用户开口）：「帮我直接点个饭」「随便给个不踩雷的」「不知道吃什么」「想吃火锅/麻辣烫/面」「20 块以内」「猫粮/纸/水是不是该补了」
- 主动：饭点 + 常点店有券 / 天气影响 / 日程显示明天有同品类局 / 采购周期到了

**核心能力 = 四类场景自动分流**

| 场景 | 触发词 | 产品形态 |
|---|---|---|
| 极度繁忙 | "没空选""直接点" | **1 个最稳一键下单方案** |
| 纠结灵感 | "不知道吃什么""给几个选择" | **4 张推荐卡**（高性价比×品质 × 稳妥×探索） |
| 明确品类 | "想吃火锅" | **常用 + 探索** 两个方案 |
| 采购补货 | "猫粮快没了"/ 周期触发 | **1-3 个补货建议** |

**关键避坑逻辑（case 张力来源）**：昨日去重（不重复推昨天吃的）、天气避坑（热天不推火锅）、日程避重（明天有火锅局今天不推）、旅游状态必出当地特色。
**安全门控**：下单前必须展示明细确认，不自动支付。

## §3 Skill 3 · 智能路线规划与分享（`route-planning-sharing` 🗺️）

**一句话**：从"查一家店"到"成局出行"全覆盖的出行助手；出行后虾蜜还会写总结、攒足迹、出年度报告。**分两个完全独立的阶段。**

### 阶段 A · 规划

**怎么被触发**：用户主动问（"周末去哪玩""想吃川火锅推荐几家""这家怎么样""怎么去最快"）／ 群聊（被 @ 全力介入，命中关键词只发一条软提示）／ 日历到点但没安排路线。

**三种粒度（按需裁剪，不一律走重流程）**
| 粒度 | 场景 | 产出 |
|---|---|---|
| 轻·单点 | 查一家店 / 算一段路 / 看一家排队 | 一句话秒回 |
| 中·单品类多店 ⭐主力 | "想吃川火锅" | 拉记忆 + 高德搜索 → **1 强推 ⭐ + 1-2 备选**（评分+人均+特点+位置） |
| 重·多节点出行 | "吃饭+看展+看电影" | `route_planner` 算 **3-4 套路路线**（最对胃口/均衡⭐/最省心/最划算），含时间可行性校验 |

**杀手锏**：重粒度会真算时间——"这么走赶不上 21:30 那场，帮你挑了 22:10 的"。所有数字来自高德真 API，可现场比对。

### 阶段 B · 分享 / 记录（纯文字为主，不依赖文生图）
- **一日份总结**：行程闭环后主动甩"今天替你总结好了，要发不？"，出两档——发群版（极短够损）+ 朋友圈碎碎念版（带日期天气头）。素材是当天真实足迹，标出"梗"（最长排队/最贵/最特别）。
- **私人足迹档案 + 年度报告**：每次出行结构化记一笔 → 攒成可回看足迹 → 到量出"望京版年度报告"（出行数/最常吃/总花费/最长排队/最常和谁，数数交脚本、叙述交虾蜜）。

> ⚠️ 文生图（即梦 imagegen）已**降级为可选、在独立分支、暂不并入主线**——写 case 时**别把 AI 漫画卡当交付物**，分享阶段的交付是虾蜜的文字 + 足迹数据。

## §4 三条最强联动（编排/服务串联评分的核心 case）

1. **Skill 3 规划出餐厅 → Skill 1 自动启动该店排队监控 + 出发提醒**（用户不用重复说）
2. **Skill 1 等位 > 30 分钟 + 饭点 → Skill 2 主动提示"先点份外卖垫一口"**
3. **Skill 3 行程结束 → Skill 2 主动问"明天要不要补今天没买到的东西？"**

> 联动只做**一次**主动衔接，不连环追问。一条贯穿三 Skill 的 case 最能体现"同一个管家在一天里不同场景主动出现"。

## §5 写 Case 的现成抓手（mock 已预埋，直接用）

**沙盒锚点**：望京 · 2026-06-07 周六傍晚。

**可用数据**：`mocks/` 里有 **50 家餐厅**（望京 42 / 三里屯 6 / 国贸 2：海底捞/凑凑/外婆家/大鸭梨/探鱼/西贝/太二/呷哺…）、**8 张券**、**1 家影院（嘉禾望京影院）**、**采购品**（猫粮 28 天周期 / 抽纸 40 天 / 无糖气泡水）、外卖菜品库 + 一份默认画像 + 今日上下文（昨日吃过/明日日程/天气）。

**预埋的关键剧本时刻（拨时间就能复现的张力点）**：
- **海底捞·望京店（shop-001）= 主角店**：18:00 后排队从 32 桌单调递减 → **18:25 跳号 -5**（前面有人放弃）→ **19:10 又涌进 +8**（突发排队潮）→ **20:00 清台 -6**。一家店就能演完"该出门没→快了→又满了→放出来"的完整弧线。
- **shop-016：18:45 突发暂停取号 → 18:55 恢复**（餐厅故障场景，演异常处理）。
- 多家店在 19:00 前后有 surge/jump/clear 事件 → 适合"并行盯多家、哪家先放出来"的对比 case。

**真实可被评委验证的**：任意望京餐厅的评分/人均/营业时间，任意两点间的距离/时长/打车费——评委掏出高德 APP 能对上。
