## 二、Nocode 案例：从 Hackathon 想法到对外产品的全过程

陈晟把整个分享凝练成一句话：**Hackathon 的起点是一个洞察，AI 工具负责把它实现。** 这段不是命题解读，但对我们后续选题非常重要——它告诉我们评委心中"好作品"的演进路径是什么样的。

### Nocode 当年怎么诞生的

时间回到 2024 年。陈晟回忆，那时 AI 在研发场景里大多还停留在"代码补全"阶段——写一段代码停在那里，AI 把后面补全。"那会儿觉得这已经是很大的进步"。这种产品形态的典型代表是 Cursor，单打独斗确实好用，但非技术同学想用 Cursor 帮自己写网页时，会被环境安装、依赖配置、代码部署这一连串技术门槛挡住。

24 年 Hackathon 的契机来了，陈晟想做的事很直白：**非技术同学只要描述自己的想法，就能直接拿到一个想要的页面，不用关心任何技术细节，还能一键部署到线上拿到分享链接。** 这就是 Nocode 最初的命题。

### 当年的团队配置 vs 今天的对比

24 年那会儿做这件事需要五人团队：组织者（陈晟本人，做任务分配、汇报材料、逻辑梳理）+ 设计师（做漂亮的原生界面）+ 前端 + 后端 + 专门调 Prompt 的同学（那会儿 AI 还不够强大，提示词需要持续打磨）。

**而陈晟反复强调一句话——"现在让我重新再做这个事情，我觉得太简单了，一到三个人就足够了"。** 他原话："你就可能去跟这个账号去对话就可以产生了，而且现在的视角来看，你可以用 AI 工具产生更好的产品。" 他甚至说"我理解 Nocode 这种形态已经落伍了"——一年半的时间，AI 时代已经从 Chat 进入到 Agent 阶段，**Agent 可以长时间稳定运行几个小时甚至一天**，只要给它一个明确任务。

> **陈晟最后那句话是这次分享的题眼**："**这里面的关键点就是你的洞察、你的想法、以及你最终想呈现出来的产品效果是什么样子。接下去你就用好 AI 工具把它实现好就行。**"

### Demo 到产品的路径

Nocode 当天的产出"现在看是不是比较 Low 啊"——陈晟自己也承认初版界面粗糙。但这套东西被陈晟丢到公司内部大群里收获了第一波天使用户，反馈不断进来推动迭代，逐渐拉入更多兄弟团队，最后演变成内部基建产品并对外发布。**这条路径不是 Hackathon 给的，但起点是一个好的想法 + 一个能跑的 demo——这就是后面所有可能性的基础。**

主持人随后总结的公式可以记下来：**真实痛点 + 核心困境 + 有限时间对想法的验证 = Hackathon 作品**。

## 三、命题 01 · 冯岩：基于 OpenClaw 的本地生活全天候私人管家

### 出题人背景

冯岩来自基础研发 ML7 部门，负责大模型训练。**ML7 的职责覆盖整个公司大模型的整体建设——LongCat 大模型的基础模型预训练、对齐、推理，以及上层 Agent 能力建设，从工具调用、长期记忆到任务编排，最后服务到各种业务场景。** 这意味着这道命题不是从单一业务部门切出来的需求，而是公司大模型基建团队对"Agent + 本地生活"范式探索的一次邀请。

### 题目背后的痛点

冯岩讲痛点很具体。**用户想"周末吃顿好的、看个展、跟朋友出去玩"，现在的实现方式是多个 App 来回跳**——点评查餐厅、地图看路线、打车软件叫车。"虽然美团提供这些功能，但你其实需要在不同的界面上来回切换，信息散落在各处。" 结果就是 **App 孤岛效应 + 用户决策疲劳**。

> 冯岩对这个问题的本质判断（这句是他原话，能直接挂钩选题逻辑）："**本质上是我们现在的本地生活服务还停留在用户主动检索的阶段，缺这样一个串联的过程。**"

OpenClaw 这类**聊天软件原生的智能体框架**带来的新可能性是——Agent 直接住在聊天软件里，在后台持续运行，通过 Skill 扩展运行机制，**从"人去找服务、串联服务"变成"服务主动去找你"**。

### 实现路径：四件事

冯岩把交付物拆成四件事：

1. **管家塑造**：通过 `.so` 文件给管家设置人设和记忆。**让它知道自己是谁、用户的习惯——不吃辣、预算两万以内、习惯坐地铁。** "这样它就不光是一个聊天机器人，它有自己的个性，能够了解用户的习惯。"
2. **Skill 开发**：本地生活相关的技能插件——路径规划、餐厅排队监控、天气和活动抓取。
3. **后台 7×24 长效运行**：用户给完需求之后管家能长效跑下去。冯岩举的例子很形象——"**我想去海底捞但现在排队，剩五桌的时候你去提醒我去叫车**"。
4. **Mock 后端**：核心环节。冯岩举例——排队接口在不同时间返回不同结果（10 分钟前有位、10 分钟后满），这种**时间事件的响应需要选手自己 Mock 一个系统的返回结果**。

冯岩反复强调一句："**核心是希望大家不只是实现一个 demo 的聊天机器人，而是利用模型的 Agent 能力去做各种任务编排、异步执行、状态监控**，让本地生活服务的体验跟现在完全不一样。"

### 评分标准（三个维度）

- **创新性**：管家人设的想象力、独特的 Skill 设计、后台协同的玩法、沙盒设计。鼓励结合大模型做差异化的交互体验。
- **功能完整性**：管家人设和记忆、Skill 是否跑通、任务演示是否能正常 run、沙盒情况模拟是否详细，包括代码结构和文档。
- **实现效果**：演示是否流畅自然、执行是否快速、后台监控结果是否可靠。**评委会现场跑真实例子做测试。**

### QA 关键澄清

- **交付形态**：聊天软件中的对话形式即可，源代码和文档以规范方式交付是加分项。
- **前端要求**：很低。Agent 产品运行逻辑主要在后端，对话即 UI，可以做得很简单——重点是功能实现的完整性和体验流畅性。
- **没有脱敏数据怎么办**：复用大模型能力做 Mock，在 Prompt 中告诉它"某些时间段会发生什么事"，**让模型去模拟各种环境**。

## 四、命题 02 · 孙一昭：复杂指令下的多轮对话评测系统

### 出题人背景

孙一昭来自履约平台技术部——美团一个比较核心的技术体系。**履约平台覆盖外卖、闪购、跑腿，从用户下单到骑手送达的全链路技术支持**——订单调度、骑手端系统搭建、运营供给保障骑手按时上线、配送时间预测、智能调度系统。

孙一昭明确点了部门正在做的"AI 数字人"三个具体场景，这些场景就是这道题的真实背景：

- **AI 站长**：定时给骑手打电话提醒上线、注意安全、参加线下培训。"**之前都是真人站长打电话，非常消耗站长时间和精力**"，现在用 AI 数字人简化这部分。
- **自动化招聘**：给求职意向者打电话问意向、走自动化招聘流程。
- **骑手助手**：商家出餐慢、配送时间晚、地图定位不准的问题，原本走客服压力很大，AI 骑手助手大幅减轻了客服压力。

### 痛点：人工评估扛不住

孙一昭把痛点讲得非常清楚。**每一通对话都会配置一个比较复杂的业务要求**——传递什么信息点、获取什么字段（用户手机号、求职意愿）。判断模型有没有满足这些要求，原本只能靠人工一个一个去检查。

人工的三大问题孙一昭挨个数了一遍：**评估耗时长、无法批量执行、没有统一指标也没有明确的优化方向。** 这是部门"现在还面临的比较大的问题"。

### 任务拆解：链路与四个关键词

整个系统链路是这样的：**任务指令 → 用户模拟器构造模拟对话数据 → 拿到模拟对话数据去做可量化评测 → 产出评估报告**。

孙一昭对每个环节都给出了明确要求：

**用户模拟器需要选手自己构造**——目标是"尽量让模拟用户能够激发出我们对话模型能够完整覆盖到任务指令里所有的情况"。这一句是隐藏的难点：模拟器既要"完整覆盖"，又要"激发出"模型的全部表现，不是简单跑几轮对话。

**过程可解释**——孙一昭原话："**之前评估的任务要求是比较长，没办法二元化区分这个任务是完成还是没完成，那这个时候很难有一个明确的优化点。**" 所以需要把对话任务**拆解得比较完整、逻辑通顺**，给出一个个评测点。

**结果可量化**——不是 0/1 分数，"是需要给出一个 0 到 100 分这么一个分数，让我们知道目前的系统能达到什么样的效果、后面的优化方向是什么"。

**一站式**——给定一个指令 + 一个待测模型，系统全链路跑，直接产出报告，**不需要人工再检查或反馈**。

### 最终交付形态（QA 里澄清的关键信息）

主持人追问"一站式平台具体什么样"，孙一昭给出了清楚答复——**有评分系统 + 前端界面，界面里用户输入任务指令，系统针对该指令做对话模拟、效果评估、最终产出可量化的评估报告**。这是一道有明确"产品形态"要求的题。

**输入数据形态澄清**：外呼数据已经从语音 ASR 转成文本，**这道题只需要考虑文本数据，不涉及语音模型**。

## 五、命题 03 · 蔡子正：美甲 AI 试戴与智能运营

### 出题人背景

蔡子正来自服务零售技术部，**支撑乐生活和亿生活两个业务线**——足疗按摩、医美美发、洗浴、亲子教育、运动健身……**涵盖近 200 个细分消费领域**，连接消费者与千万级本地服务商家。部门在 C 端、B 端、H 端（销售端）都有 AI 项目：

- **C 端**：懂需求、会推荐、能完成预约交易任务的智能助手。
- **B 端**：数字员工帮商家自动上单、做经营数据分析，提高经营效率。
- **H 端**：AI-BD 辅助系统端到端完成日常销售任务。

### 痛点：两个看不见

蔡子正把美甲这个场景的核心矛盾归纳为**"用户没办法实时验证效果，运营没办法实时感知市场"**——两个看不见。

**用户侧**：在美团上浏览美甲团购最大的障碍是**想象成本极高**。"用户浏览了非常多美甲图片，但很难脑补出来这个图片戴在自己手上到底好不好看——肤色搭不搭、手型配不配。" 到店试戴成本又高。**无法所见即所得 → 决策周期长 → 放弃率高**。

**运营侧**：千万级款式库存 + 流行趋势瞬息万变 + 主要靠人工统计 → **更新滞后**。蔡子正举了一个非常具体的例子："**我们发现小红书上有些款式比较火，但等我们识别到这个款式已经火、把团购上上去的时候，往往已经错过了最佳营销窗口。**" 这一句直接揭示了运营侧对"实时性"的本质需求。

### 两个核心能力

蔡子正把任务拆成两个能力：

**用户端 · AI 试戴**：用户上传手部照片 + 选择心仪款式，系统把款式**高保真还原到用户手上**。算法上有两个具体要求：
- **款式特征要准确还原**：颜色、图案、纹理、风格不能走样。
- **生成图片与用户上传的手图保持高度一致**：原本的肤色、光影、手指姿态和纹理都要完整保留。

**运营端 · 款式智能运营助手**：蔡子正这里直接点名"**推荐大家使用像 WinkStudio 这样的 AI 助手**"——让运营系统具备自动识别爆款趋势的能力，小红书上一个款式火了，能自动快速识别并上架到试戴应用；同时给冷门款式预警，做自动下架处理。

### 评分标准（四个维度）

- **完整性**：功能完整、流程清晰、代码结构合理、质量高、系统稳定运行。
- **创新性**：素材算法、运营 Agent 算法、技术实现、产品设计有新颖做法且能带来效果与效率提升。
- **应用效果**：素材效果如何、运营策略是不是真正有效。
- **商业价值**：是否具备落地潜力、规模化应用价值，能否带来用户增长、效率提升、成本优化。

### QA 关键澄清

- **运营侧热度监控范围**：最好能延伸到 UGC 社区，**如果实现成本高也可以自己 Mock 数据**。
- **交付形态**：需要完整 demo + 可提供试用入口（**部门有专业产品同学来体验和评测**），能提供代码仓库最好。

## 六、命题 04 · 杨浩林：AutoSolver——AI Agent 自主求解配送分配

### 出题人背景

杨浩林来自 KeeTa 技术团队配送算法组。**KeeTa 是美团旗下专注国际市场的配送平台**，2022 年 10 月成立，已进入香港、沙特、巴西市场。KeeTa 下设独立完整的技术部门（前后端、测试、风控），算法部门包括大模型应用、营销定价、搜推算法等。**杨浩林所在的配送算法组负责骑手订单实时调度、配送各环节时间预估、骑手定价激励、LBS 以及供需调节。**

### 题目的精神——一句话定基调

杨浩林一上来就把整道题的精神拎了出来——**"这不是一道让你写出最优算法的题，而是让你构建一个能自己找到最优算法的 Agent。"** 他强调"**这个区别非常关键。你的目标不是解这一道题，而是要造一个会自己解题的系统**"。

他最后还补了一句更狠的："**一个笨但是会自我进化的 Agent，要比一个聪明但是固定的算法更符合本题的期待。**"

### 真实业务场景与复杂性

问题来自外卖真实业务——**每天高峰期系统需要在几秒钟内把几百个配送订单分配给附近骑手**。听起来简单，实际上有几层复杂性叠加：

- 两个相邻订单可以**合并派给同一骑手**（合单），也可以分开派给两个骑手。
- 各地区管控政策不同，骑手不一定会接单——**系统可以预估骑手接单概率**。
- 派单时**可以把同一订单同时派给多个骑手，谁先接谁送**。
- **同一轮次每个骑手只能被指派一个订单或一个合单包**。

这些因素组合后**得到了一个 NP-hard 的组合优化问题**。

### 输入与输出

**输入**：一张候选表，每一行代表"某骑手接某订单或合单的分数 + 接单概率"，分数评估骑手完成该订单的好坏。订单可能是一个任务单，也可能是两个任务单组成的合单包。

**输出**：一份分配方案列表，每条记录说明哪些订单分配给哪些骑手。

**示例代码已通过邮件发给报名选手**，注意查收邮箱。

### 评分函数与策略权衡

评分用一个总分函数，**未分配订单有一定惩罚分数**——所以策略上必须考虑骑手接单概率。

杨浩林给出了一个具体策略选择上的权衡："**如果一个订单骑手接单概率很低，你可以同时派给多个骑手，由概率决定谁来配送，这样降低订单没派下去的风险，但会消耗更多骑手资源。**" 这种取舍正是本题的核心趣味之一。

### Agent 的四件事

Agent 系统需要能做到：

1. **自主探索策略**：Agent 自己提出"我要试试贪心、我要试试局部搜索"，**不需要人来告诉它用哪个方法**。
2. **自动评估筛选**：每一轮判断当前策略是否比上一轮更好，决定保留还是丢弃。
3. **迭代改进**：根据历史实验结果调整优化方向——比如"**发现贪心在合单上表现差，就自动转向其他策略**"。
4. **输出最优解**：给出当前最好的分配方案。

> 杨浩林强调："**评委想看到的是你的 Agent 在多轮迭代之后，确实比第一轮要有很大的提升。**"

### 评分构成（三段）

- **50 分**：最终解的质量。
- **30 分**：Agent 的自主迭代能力。
- **20 分**：技术报告。

**评价网站排行榜只占一半**，另一半在技术报告。技术栈不限制，**但因评价网站限制，语言限定 Python，且服务器 Python 版本较低，要限制到 Python 3.6 以前**。

### QA 关键澄清

- **算法与 Agent 开发的比重**：杨浩林没给具体比例，只说"算法本身看解的质量就可以了，还有一半评价看报告的写作水准"。
- **10 秒时间限制**：**每个算例的求解耗时上限是 10 秒**。

## 七、命题 05 · 一添：本地路线智能规划

### 出题人背景

一添来自**点评事业部，是点评的 AI Product 产品经理**。她自我介绍是"重度的点评用户 + 资深吃货 + 产品经理"——这道题"某种意义上也是我作为一个资深吃货和产品经理想跟大家解决的一个真实的痛点"。

点评的整体盘子值得记一下：**每天服务数亿用户的吃喝玩乐决策**，海量星级评价内容，**百万级 POI、亿级 UGC 评价**，加上丰富的用户线下行为偏好数据。

一添对 AI 团队当下在做什么有一句很精准的总结：**"让 AI 帮你从找到信息进化到直接帮你去做好决定。"** 用 POI + UGC + 偏好数据 + 大模型，帮用户省掉纠结、比价、做攻略的苦差事。

### 真实场景：从一个亲身经历讲起

一添讲了一个完全真实的故事——**上个月有朋友从外地来上海，让她安排一天行程，"听起来简单的命题，实际操作非常复杂，花了我大概半个小时"**。

她原话还原了整个步骤：在点评搜静安寺附近的 brunch，翻十家收藏三家；再搜附近好玩好逛的地方翻一堆；朋友反馈下午三点有个工作电话要占一小时，**要重排这段时间**；过程中还要算两个地方打车多久、网红店周末是否要排队、要不要提前预定、预算够不够。

> 她由此引出一句话定调："**用户每一次出行，他真实的数据并不是点位的顺意，而是一套真实的路线。**"

### 任务拆解：四步核心路径

一添把任务路径拆成四步：

1. **意图理解**：用自然语言意图表达，深度理解。
2. **数据匹配**：理解意图后根据 POI 数据、评分、营业时间等基础信息匹配。
3. **路线生成**：自动生成一条**真正合适的路线**——"不是随便拼接几个高分餐厅，而是真正考虑时间、距离的衔接，有一个完整、匹配的动线"。
4. **动态调整**：**这一步一添特别强调"是我们特别看重的"**。系统要有容错、能跟用户在线交互——比如用户反馈"那个咖啡馆评分太低了换一个"，系统要能实时响应、互动，**而不是从头再规划**。

### 评审三维度

一添把评分拆得很清楚：

**完整性（基本盘）**：路线是不是真的拿来就能用。"**中午安排的餐厅 11 点才开门、两个景点之间换乘一个小时——那这种就不行**"。要现实、要靠谱、能处理用户主动表达的约束条件和冲突。

**创新性**：两个层面——
- **体验创新**：让人眼前一亮的交互，比如方便地调整路线或多方案对比，发挥创造力做出惊艳亮点。
- **技术创新**：大语言模型和检索规划怎么融合、多 Agent 协作能力等。

**应用效果**：综合工程能力——代码清不清楚、能不能流畅跑起来、解释全不全。**工程实践足够好、代码可维护性强，会有额外加分。**

### 三个面向用户场景的硬约束

一添最后拎出三个硬约束（这部分会决定架构设计）：

1. **响应时间 10 秒以内**——一添补了一句关键澄清："**这个 10 秒的时间是实际 token、首 token 出的时间**"。她还补了实操建议：架构上可以**并行调用、缓存设计、流式输出让结果更快**；交互上不要让用户傻转 10 秒看不到任何东西，**loading 体验也要流畅**。
2. **POI 类型至少覆盖两大类**——餐饮 + 其他（娱乐文化等）。
3. **路线至少串联三个 POI**。

### QA 关键澄清

- **是否允许多轮对话**："**当然是 open 的，实际用户场景就是会不断调整、接近用户意图。**"
- **交付形态**：**完全不限**。APP、小程序、其他形态都行，"只要用户能够用，不管什么形式都 OK"。

## 八、命题 06 · 刘璐：周末闲时活动规划

### 出题人背景

刘璐来自**美团平台的小团 Agent 产品部**。小团这款产品当下在做的事是——**基于美团下用户的"找东西、找服务、找信息，包括做一些规划"的场景**。刘璐明确点出："**当前这个命题也是目前美团用户在当前小团里面比较高频热门的一个痛点问题，所以大家去解决的是一个真实用户场景。**" 也就是说，这道命题就是小团团队正在做的事，参赛作品可能直接进入产品迭代。

### 痛点：周末怎么过好

刘璐把痛点描述得比较朴素——"**周末好不容易休息，要怎么样去规划自己的闲时**"。可能跟朋友约、可能一个人逛、可能和家人多人聚会。她总结的目标是——**既不要像特种兵一样紧张，又能闲适地优化、充实生活**。

任务定位：**不同用户场景、不同用户人群下，帮他把周末行程安排好**，做周边或城市周围的短途旅游和任务规划。

### 用户层面的产品形态

用户要做的事很轻——"**通过一句话的语言，能够完成一个整体的解决方案**"。这个方案可以是：

- 路线串联
- 活动的具体规划
- 结合吃喝玩乐的综合场景串联（热门打卡、周边游等）

### 交付要求

**交付物规格完全不限制**——Webview、小程序、APP、命令行、多轮单轮对话都可以。目标是把场景内容**展示完整、展示清晰**。

**Mock 政策**：用户画像、个人信息、场景信息**不限于使用 Mock API 或 Mock 数据**，保证一个比较饱满的完成度。

**设计说明书**：**小于两页**的设计思路和说明，补充 demo 或 coding 之外的内容、凸显选手想设计的重点功能和场景。

### 评分维度

**完整性**：完整解决命题；满足不同人群、不同地域的要求；**结果要符合自然规则、时间分配合理**，形成真正可实操可落地的任务；不管走 Mock API 还是真实外部网络数据，**保证真实精准调用**，跑通端到端调用成功率，**不要出现兜底和超时过高的情况**。所以设计中需要具备**异常处理能力**、兜底策略和设计。

**技术创新性**：通过 Coding 和这些能力的创新性。**充分思考不同用户在周末的不同诉求**，做个性化的思路设计和排版，形成创新功能。UI 交互上 Chatbot 或 Web 形式不限，目标是让用户更直观地选择方案。

**加分项与底线约束**：
- **加分项**：UI 足够结构化；用 Mock API 能完成高质量、高完整度的任务。
- **底线**：对生成速度和兜底速度有要求——保证完整任务跑通、完整效果展示。

### QA 关键澄清

- **和美团小美 App 的关系**：刘璐明确——"**这个赛题和小团还是小美关联性并不是很强**。不管是小团还是小美，包括点评，都是大家在日常生活中真实用户所遇到的问题和需求，是他们都会存在的共性。这次赛题主要也会围绕解决用户真实需求的出发点。" 所以参赛作品**不需要绑定到具体某个 App 形态**。
- **目标人群范围**：题面里给的"三口之家"和"情侣"只是示例，"**完全 open**，可以根据日常或对周围朋友、家人的分析，尽可能覆盖足够多的用户场景"。

## 九、自由赛道说明

**核心理念**：本地生活全场景的自由探索。**工具有强制要求**——选择自由赛道的团队必须使用 **Nocode 或 CapCode 其中一种**：

- **Nocode**：陈晟的优秀案例同款。零编程背景、自然语言对话生成网站和小程序的平台，**可创造互动网站、提效工具、企业管理系统**等应用，实时预览、局部定位修改、一键发布分享。

## 十、为后续选题分析准备的几个观察

整理过程中浮现的几条信息——它们不是评委的明文标准，但是从出题人话术里反复透出的隐含偏好，**值得在选题决策时挂回来对照**：

**评委的核心关注是"AI 解决真实业务"，不是技术原创性**。主持人开篇就锚定了，"AI 和真实业务的碰撞"是大赛最大亮点；陈晟收尾说"你的洞察、想法、最终产品效果是关键，AI 工具负责实现"；几乎每个出题人都强调"真实场景""真实痛点""真实用户"。

**六道命题在"PM 友好度"上的分层从出题人话术里就能读出来**：
- 冯岩、刘璐多次强调"用大模型 Mock"、"形态完全不限"、"前端要求低"——这是给非算法/非工程团队留的口子。
- 蔡子正、杨浩林的题面里"算法"、"高保真还原"、"50 分看解的质量 + Python 3.6 + 排行榜"等表述，明显是给算法 + 工程混合团队的。
- 孙一昭、一添介于中间，工程要求中等，但有"一站式平台 + 前端界面"、"代码可维护性强会有额外加分"这种工程门槛。

**"产品形态明确度"差异**：
- 命题 02 明确要求"前端界面 + 一站式平台"——产品形态被锁死。
- 命题 03 要求"完整 demo + 试用入口 + 部门专业产品同学评测"——形态偏完整产品。
- 命题 01、05、06 都是"形态不限"——CLI 也行、Web 也行。

**数据策略要分清**：命题 2、3、4 拿脱敏数据；命题 1、5、6 自己 Mock。**Mock 政策被冯岩、刘璐多次主动确认**，意味着这三道题在数据层面对选手是友好的。

**"Mock"在命题 01 是一个隐藏工作量**——冯岩说的"不同时间返回不同结果"的状态机 Mock 比一次性 Mock 接口复杂得多。这一点选题时要算进去。

**10 秒约束的两种解读**：
- 命题 04 是**严格的求解算法耗时**（每算例 10 秒上限）——硬约束、技术性。
- 命题 05 是**首 token 时间**——一添明确澄清，允许并行/缓存/流式来达成，且交互上可以用 loading 体验弥补。**这两个 10 秒不是一回事，不要混淆。**

**陈晟分享里那句话可以直接作为我们做选题决策的锚**——"**现在让我重新做这个事情，一到三个人就足够了**"——AI 工具的进步使三人 PM 团队完成一个像样产品的可能性，比 24 年高了很多。但**前提是"洞察 + 想法 + 最终产品效果"清晰**，不是堆功能。
