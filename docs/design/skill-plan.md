# 0531 · 3 Skill 方案 v6（产品定义锁定版）

> **v6 vs v5 关键变化**：
> - **Skill 1**：不变（4 条入口路径已锁定）
> - **Skill 2**：由"算账助手"换为"**智能外卖与采购助手**"——整合外卖 + 小象超市，主动+被动双模式
> - **Skill 3**：由"订单守护"换为"**智能路线规划与分享**"——基于 [`skill3-feasibility.md`](skill3-feasibility.md) 确认走主动推荐型，接真高德 API
> - 新增"Skill 候选储备"节，保留顺路助手等不主推方向

---

## 总览

| Skill | 一句话定义 | 管家何时主动出击 | 核心交付 |
|---|---|---|---|
| **1. 多店并行排队管家** | 用户给候选餐厅或管家自己注意到→并行盯排队、自动取号、临场提醒 | 用户表达 / 用户美团订座 / 习惯饭点 / 用户曾提过的店出现窗口 | 后台监控 + 自动取号 + 出发提醒 + 联动叫车 |
| **2. 智能外卖与采购助手** | 主动给用户推今天该吃啥/买啥，或用户问时秒出最优解 | 饭点 + 常点店有优惠 / 季节促销 / 用户主动问 | 一键下单方案（外卖 + 小象超市） |
| **3. 智能路线规划与分享** | 群聊讨论→管家自动出行程；行程后→管家做漫画卡片 | 群聊关键词触发 / 用户日历到点 / 用户主动问 | 真行程文档（接高德）+ 分享卡片（Canvas 渲染） |

**共同哲学**：管家**主动观察用户状态**（订单、习惯、群聊、行为）并出击；数据**80% 真（高德 / wttr.in / 美团订单结构）+ 20% 合理 Mock**（排队/券/票务运营层）。

## Skill 1 · 多店并行排队管家

### 用户需求

**痛点**：
- 想吃某家餐厅但不知何时该出门 → 错过取号窗口
- 多家备选时无法同时盯 → 只能赌一家
- 排队信息更新不及时 → 提前出门白等 / 到了发现已过号
- 高峰期想吃但不甘心干等
- 美团上订座了，但临场排队情况不知道

**真实使用场景**：
- 周末聚餐、商圈逛街、午餐高峰、纪念日订座
- 多店选择困难
- 商务接待提前确认餐厅就位
- 已订座但担心实际入座等多久
- 周五下班路上还没决定但通常会出去吃

### 基本逻辑设计

**4 条入口路径**（同等地位）：

| 路径 | 入口方式 | 范式 |
|---|---|---|
| A | 用户主动表达："今晚吃 X 或 Y" | 用户起意，管家承接 |
| B | 管家观察到用户在美团 APP 订座 → 自动启动该店排队监控 | 服务找人 |
| C | 管家基于习惯主动出击：周五 17:00 + 用户每周五都外出吃饭 → "今晚想去哪？要不要先盯一下排队？" | 服务找人 |
| D | 管家基于事件主动出击：用户曾提过的店今晚正常出窗口 → "您说过想去那家店，今晚 7 点估计能排到，要现在帮您盯吗？" | 服务找人 |

**触发后的统一动作**：
1. 后台并行监控所有候选店实时排队（Mock，但和虚拟时钟同步）
2. 综合排队 + 用户当前位置 + 偏好 → 推荐排序
3. 达到取号阈值 → 自动取号（必要时先问用户确认）
4. 持续追踪前方桌数
5. 临近到位 N 桌 → 主动推送出发提醒（含到店建议路径，调高德路径规划）
6. 联动叫车 Skill

## Skill 2 · 智能外卖与采购助手

### 用户需求

**痛点**：
- 每天饭点挑外卖都是重复劳动，挑半天眼花
- 常吃店出新券 / 限时优惠没注意，错过白错过
- 多店多券规则算不清能否叠加；凑单反直觉没耐心算
- 日用品（水、纸、调料）想起来已经空了
- 季节变化（如夏天想要冰饮、冬天想要热饮）的偏好微调用户自己不会主动调整
- 多人订餐凑单更复杂

**真实使用场景**：

> 外卖
- 每天午晚饭点固定到 → 想要"刚好"的方案不用动脑
- 常吃店当天有优惠 → 不主动搜不会知道
- 在外排队等位很久 → 想先点份外卖垫一口
- 多人订餐 → 凑单算账复杂

> 日用品采购（小象超市等）
- 家里水/纸/调料用完了懒得专门下单
- 季节性需求（如换季囤被子/驱蚊液）容易忘
- 周期性消耗品（如猫粮、宝宝奶粉）忘了补货

### 基本逻辑设计

**两种模式同时存在**：

**主动模式**（管家主动出击）：
1. 基于用户**历史订单数据**：常点菜品、用餐时间、是否偏好套餐、采购周期
2. 加上**特定情境**：季节变化 / 限时促销 / 库存预警（"上次买猫粮 23 天前，估计快没了"）/ 节假日
3. 命中触发条件 → 推送"刚好"的方案，含**一键下单**链接
4. 推送话术示例：
   > "您每周三这个点都吃 X 家猪脚饭，今晚他家神券 6 折，凑过满减实付 ¥28。要现在帮您下单还是 7 点？"
   > "上次买猫粮 25 天前，按上次的量估计后天就没了。小象超市今天满 50 减 10，凑 2 包 + 罐头 ¥58 减后 ¥48，要下单吗？"

**被动模式**（用户主动问）：
1. 用户表达品类/约束意图（"今晚想吃猪脚饭"/"找个 20 块以内的午饭"）
2. 拉附近候选店 + 当前可用券池 + 凑单方案
3. 输出 top 1-3 性价比最优方案，含店名/菜品/用什么券/实付/配送时间
4. 用户追问处理（"再算 X 家" / "换品类" / "今晚要辣的"）

**Scope 明确**：
- ✅ 外卖（高频、SKU 多、可比性强）
- ✅ 小象超市等日用品（采购周期感知，"懒得选"的延伸）
- ❌ 堂食（SKU 单一无对比意义）
- ❌ 到店服务（美甲/美发 SKU 不可比、体验主观）

## Skill 3 · 智能路线规划与分享

> 技术可行性详见 [`skill3-feasibility.md`](skill3-feasibility.md)——本节给产品定义。

### 用户需求

**痛点**：
- "周末吃顿好的、看个展、跟朋友出去玩" 现在要在多个 APP 之间来回跳（点评查店、地图看路、打车叫车、票务订座）→ **APP 孤岛 + 决策疲劳**（冯岩原话）
- 群里讨论出去玩，没人愿意做"行程秘书"汇总
- 自己规划要权衡多重因素（偏好 / 距离 / 营业 / 排队 / 票务 / 天气）→ 太累
- 行程结束想发朋友圈但素材散落各处，没耐心整理

**真实使用场景**：

> 规划阶段
- 周末和朋友出去玩（吃饭+看展+看电影+逛街）
- 与家人/对象的小型出游
- 商务接待安排（餐厅 + 路线 + 时间控制）
- 群聊讨论出去玩但无人愿意做统筹
- 用户日历到点（如"周六晚 7 点和朋友吃饭"）但还没安排具体怎么去

> 分享阶段
- 出游结束想发朋友圈但找不到素材整理
- 想给参与的朋友/家人一个"今日小结" 留念
- 想做有创意的卡片（不是普通九宫格）

### 基本逻辑设计

**两个阶段独立但相连**：

#### 规划阶段（管家做"行程秘书"）

**3 种入口**：
- A. 管家在群聊里被 @ 或自动捕捉到关键词（"周末"/"出去玩"/"去哪") → 实时整理讨论中的决定
- B. 用户日历到点（如"周六晚 7 点和朋友吃饭"）但未安排路线 → 主动出现："要不要我帮你做行程？"
- C. 用户主动问"周末去哪玩好"或类似

**核心动作**：
1. **拉用户偏好**（自动）：从 OpenClaw memory 读取饮食偏好、出行习惯、朋友信息
2. **拉候选 POI**（接高德真 API）：周边搜索符合主题的地点，拿评分 / 人均 / 营业时间 / 商圈 / 特色 / 电话 / 图片
3. **算路线**（接高德真 API）：起点→各候选点的距离 + 时长 + 步行/驾车/打车/公交多种方式 + 真实出租车预估费用
4. **拉天气**（接 wttr.in 真 API）：当日是否下雨/温度，影响推荐（如下雨优先有遮阳/有停车的店）
5. **叠加 Mock 业务层**：排队预警（复用 Skill 1 数据）/ 票务库存 / 当日优惠
6. **LLM 生成行程**：基于上述真数据 + Mock 数据，输出**多节点行程文档**：

```
🗓 周六晚
1. 19:00 「赵小姐的虾」（簋街）⭐ 强推
   🍽 4.7 分 · 人均 ¥85
   ⛽ A 从东直门 1.2km / 走 15 分钟
   ⛽ B 从国贸 6.8km / 打车 ¥22 / 22 分钟
   ⛽ C 从望京 9.5km / 打车 ¥28 / 28 分钟
   ⏰ 建议 18:45 前到（当前排队 25 桌，约 30 分钟入座）

2. 21:30 「嘉禾簋街影院」《XXX》
   🎬 步行 200m / 3 分钟
   🎟 还有 4 个连座
   💡 提示 21:00 前到，避免错过开场

要我帮你订餐厅 + 锁电影票吗？
```

**关键保证**：所有数字（距离/时长/评分/人均/营业时间/打车费）**全是真高德 API 返回**，评委可以现场用高德 APP 比对验证。

#### 分享阶段（管家做"漫画风格小结")

行程结束后（或用户主动召唤）：
1. 拉今日**实际足迹**（用户去过的店 + 时间）
2. Canvas 面板渲染 **卡通风格 HTML/CSS 卡片**（不是真 AI 漫画图，但视觉风格统一）：
   - 顶部 SVG 简笔风格地图，画出今日去过的点
   - 中间打卡时刻 emoji + 时间标签
   - 底部 "今日总结" 一句话文字
3. 调 `canvas.snapshot` 截图导出
4. 通过 Telegram/Feishu 把图发给用户
5. 用户**直接拿图发朋友圈**

**demo 时双输出**：聊天里管家说话 + Canvas 面板同步弹出图片 → 评委同时看到"对话"和"视觉"两个交付。

## 三个 Skill 的联动剧情

**核心 framing**：3 个 Skill 不是独立功能，是同一个管家在用户一天里不同场景主动出现。

**3 条最强联动**：
1. **Skill 3 行程规划** → **Skill 1** 自动启动该餐厅排队监控
2. **Skill 1 等位 > 30 分钟** + 饭点 → **Skill 2** 推荐外卖垫一口
3. **Skill 3 行程结束** → **Skill 2** 主动询问"明天要不要补一下今天没买到的东西？"

## Skill 候选储备（不主推但保留）

> 团队后续如果有时间余裕，可从这里捞回来做；也展示我们的设计探索。

| 候选 | 为什么暂时不主推 |
|---|---|
| **顺路助手 / 订单守护管家** | Skill 范围与 Skill 1 排队监控 + Skill 3 路线规划有重叠；充电宝场景虽然 demo 高光但单独成 Skill 显得功能太单点 |
| **AI 找店 + POI 深度探索** | "意外发现"价值高但 demo 30 秒讲不清；地图探索的视觉冲击需要前端工程量 |
| **Flipbook 风格 3D 可交互地图** | 视觉效果惊艳，但 token 燃烧器（生成成本高） + 复杂前端实现 → 不在 hackathon 时间内 |
| **临场救火管家** | 替平台兜底，评委是平台方逻辑反过来；用户经验中低频 |

## 给团队最后对齐的 4 个决策点

1. **Skill 1 入口扩展**：B/C/D 三条主动入口同等地位，还是分主次？
2. **Skill 2 主动模式骚扰风险**：每天饭点都推会不会烦？要不要设置"用户接受 3 次推荐后才开启"等门槛？
3. **Skill 3 群聊触发灵敏度**：默认 `@` 才回，还是关键词主动介入（"出去玩"/"周末"/"去哪"）？
4. **3 条联动 demo 全展示，还是挑最强的 1-2 条**？
