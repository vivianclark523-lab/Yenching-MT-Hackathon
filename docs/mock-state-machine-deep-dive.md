## 一、先纠正一个常见误解：Mock 不是"假数据"

很多人第一次听到 "Mock" 这个词，理解为"假数据"——写一份 `restaurants.json`，里面 5 家店各种字段。

**这个理解只对一半**。

在 hackathon 这个场景下，Mock 的真正含义是：**假装有个真后端**——不是一份死的数据文件，而是一段**模拟真服务器行为**的代码。

### 关键差别

| 维度 | 死数据（JSON 文件） | 真后端（活的）| Mock 状态机（我们要做的） |
|---|---|---|---|
| 18:00 查海底捞排队 | 30 桌 | 30 桌 | 30 桌 |
| 18:30 查海底捞排队 | **还是 30 桌** ❌ | 20 桌（队伍前进了）✅ | 20 桌 ✅ |
| 19:00 查 | **还是 30 桌** ❌ | 8 桌 ✅ | 8 桌 ✅ |

真实美团服务器返回的数据**随时间变化**。我们的 Mock 必须**模拟这个变化**。

一份死 JSON 文件做不到。**Mock 状态机就是"用代码模拟这个变化"的工程模式**。

## 二、什么是"状态机"？为什么用它做 Mock？

### 2.1 状态机的本质

状态机 = "**系统的当前状态** + **状态如何根据输入变化的规则**"。

在我们的场景里：

- **当前状态** = "X 餐厅现在排队 N 桌"
- **输入** = 时间 t
- **规则** = 随着时间推进，N 应该如何变化

把这三件事写成代码：

```python
def restaurant_queue(shop_id, t):
    """根据时间 t 计算 shop_id 餐厅当前排队多少桌"""
    if shop_id == "shop-001":  # 海底捞
        if t < "17:00":
            return 0  # 还没到饭点
        # 17:00 后每分钟队伍前进 0.5 桌
        minutes_elapsed = (t - "17:00").minutes
        return max(0, 35 - minutes_elapsed * 0.5)
```

### 2.2 这段代码的 4 个关键性质

1. **没有数据库读写** —— 没有任何"存储"的概念，每次调用都是**纯计算**
2. **同样的 t 永远返回同样的值** —— **可复现**
3. **t 改变，结果改变** —— **时间敏感**
4. **多个 Skill 同时查，看到的都是同一个值** —— **一致性**

这就是状态机 Mock 的本质：**用函数代替数据库，时间是这个函数的核心参数**。

### 2.3 为什么这种"用函数代替数据库"的设计是对的

因为我们要模拟的真实世界本来就是"**时间驱动的连续变化**"：
- 排队队伍随时间前进
- 充电宝租金随时间累加
- 神券有时段限制
- 商家有营业时间

这些都不是"存进数据库的事实"，而是"**关于时间的函数**"。**用函数表达本来就是模拟现实最自然的方式**。

## 三、为什么"动起来的 Mock"是评分要求

回到冯岩直播原话：

> "排队接口在不同时间返回不同结果（10 分钟前有位、10 分钟后满）。**这种时间事件的响应需要选手自己 Mock 一个系统的返回结果**。"

为什么冯岩特别强调这一点？

因为这道题的核心评分点是"**Agent 能力：任务编排、异步执行、状态监控**"。这三件事都需要"**世界在变**"才有意义：

| Agent 能力 | 为什么需要 "动起来的 Mock" |
|---|---|
| **任务编排** | 管家 17:30 启动监控，18:25 决策取号 —— 没有时间变化就没有"决策的时刻" |
| **异步执行** | 管家不一直跟用户说话，但在合适时机出现 —— "合适时机"由状态变化定义 |
| **状态监控** | 管家持续读后端状态，发现关键变化 —— 状态不变就没什么可监控 |

如果后端永远不变，Agent 这三件事都演示不出来。**Mock 不动 = demo 没有灵魂**。

### "动起来的 Mock" 怎么实现？三个候选方案

| 方案 | 优点 | 致命缺点 |
|---|---|---|
| 用**真实时间** + **真随机数** | 看起来真 | demo 现场无法控制何时发生什么 → 关键瞬间错过 → 录视频要拍 50 次 |
| 让 **LLM 实时生成** Mock 状态 | 灵活、不用写代码 | 同一时间问两次得到不同答案 → 评委拨时间穿帮 |
| **预定义函数（状态机）** | 可控 + 可复现 + 时间敏感 | 写代码稍麻烦，但**唯一可行** |

**所以选状态机不是技术品味，是评分要求逼出来的**。

## 四、三种状态机类型 + 业务直觉

我们项目 90% 的业务数据都能用这三种之一（或组合）来 Mock。

### 4.1 单调推进型 —— "一个方向单调变化的量"

**业务直觉**：
- 排队队伍只会越来越短（不会越来越长，除非店关门）
- 充电宝租金只会越来越贵
- 票务库存只会越来越少
- 用户位置在路上移动（距目的地距离单调减少）

**数学**：线性递减/递增

```python
state(t) = max(0, initial - rate * (t - t_start))
```

或对应的递增版本：

```python
state(t) = initial + rate * (t - t_start)  # 如计费累加
```

**适用我们项目的具体场景**：
- Skill 1：餐厅当前排队数（递减）
- Skill 2：神券剩余库存（递减）/ 配送时长（接近送达时递减）
- Skill 3：充电宝累计费用（递增）/ 用户到目的地剩余距离（递减）

### 4.2 时段周期型 —— "按时间分段周期波动的量"

**业务直觉**：
- 餐饮高峰是 11-13 点和 17-19 点
- 神券限时段（如每天 18:00-19:00 开放）
- 周末打车贵 / 工作日早晚高峰贵
- 影院黄金场是周末晚上 19:00-21:30

**数学**：分段函数 / 余弦曲线

```python
def state(t):
    hour = t.hour
    if 11 <= hour <= 13: return "高峰"
    if 18 <= hour <= 20: return "高峰"
    return "平峰"
```

或带具体倍率：

```python
def coupon_active(t):
    if 18 <= t.hour < 19:
        return {"available": True, "discount": 0.6}  # 神券 6 折
    return {"available": False}
```

**适用我们项目的具体场景**：
- Skill 2：神券限时段（"18-19 点 6 折"）/ 配送费时段倍率
- Skill 3：高德实时路况（"早晚高峰拥堵"）/ 影院黄金场票务紧张度

### 4.3 事件触发型 —— "突发的、不可预期的事件"

**业务直觉**：
- 今天 18:45 海底捞冒出"设备故障"消息
- 今晚 19:30 突然下大雨
- 某商家临时上架神券
- 前面有人放弃排队 → 队伍跳号

**数学**：事件列表 + 时间触发

```python
events = [
    {"time": "18:45", "target": "shop-001", "event": "queue_jump", "delta": -3},
    {"time": "19:30", "target": "city-bj", "event": "weather", "value": "暴雨"},
]

def queue_jumps(shop_id, t):
    """返回 t 之前所有针对 shop_id 的事件累计影响"""
    relevant = [e for e in events if e["time"] <= t and e["target"] == shop_id and e["event"] == "queue_jump"]
    return sum(e["delta"] for e in relevant)
```

**适用我们项目的具体场景**：
- Skill 1：餐厅突发故障 / 队伍跳号 / 店家提前关门
- Skill 2：限时神券突然上架（"刚才 X 店冒了张 5 折券"）
- Skill 3：交通事故 / 暴雨突至 / 影院临时换厅

### 4.4 三类组合 = lifelike Mock

真实业务大多是三类组合。**例**：海底捞当晚排队的完整 Mock：

```
基础：单调推进型      17:00 开始 35 桌，每分钟 -0.5
叠加：时段周期型      18-20 点高峰，rate 减半（队伍前进慢）
叠加：事件触发型      18:45 注入"前面有人放弃"事件 → 跳号 -3
```

三层叠加 = 一个**非常 lifelike** 的排队曲线，完全用纯函数表达。

**任何 t 调用都得到确定值**，评委拨到 17:30 看到一个数，拨到 18:50 看到另一个数，拨回 17:30 看到的还是第一个数。

### 4.5 多样化通过参数实现 —— 代码通用，参数驱动

读到这里你可能会想：餐厅各家排队速度不同、高峰时段不同、突发情况也不同。**难道每家店都要写一份代码？**

**不需要**。

**关键洞察**：3 种状态机类型是**通用代码模板**——所有店铺、商品、用户共用同一份实现。**多样化全部由 JSON 参数表达**。

#### 实例：3 家火锅店各自的"性格"

同一份 `mocks/restaurants.json` 里：

```json
"海底捞王府井": {
  "start_at": "17:00",
  "initial_queue": 35,
  "rate_per_minute": 0.5,
  "peak_modifier": {"17-19": 0.3},
  "events": [{"time": "18:45", "type": "queue_jump", "delta": -3}]
},

"巴奴毛肚火锅": {
  "start_at": "16:30",         // 开门更早
  "initial_queue": 50,         // 初始队更长，更火
  "rate_per_minute": 0.3,      // 队伍前进更慢
  "peak_modifier": {"17-21": 0.5},  // 高峰窗口更长
  "events": [{"time": "19:00", "type": "fault", "reason": "桌不够暂停取号"}]
},

"凑凑火锅": {
  "start_at": "17:30",
  "initial_queue": 10,         // 冷门
  "rate_per_minute": 1.0,      // 前进快
  "events": []                 // 无突发
}
```

**3 家店 3 种性格**：

| 店 | 性格 | 由哪些参数表达 |
|---|---|---|
| 海底捞 | 标准火爆，中途跳号 | initial 中 + rate 中 + 1 个 event |
| 巴奴 | 最难排（人多 + 慢 + 高峰长 + 突发暂停） | initial 最大 + rate 最小 + peak 最长 + fault event |
| 凑凑 | 冷门（短队 + 前进快 + 无突发） | initial 最小 + rate 最大 + 无 events |

#### 为什么这个设计如此强大

1. **加新店 = 改 JSON，不动代码** —— 评委说"如果初始排队 100 桌会怎样" → 改一个数字。加第 4 家"网红日料店" → 加一段 JSON。加 50 家、500 家都是同样的代码
2. **跨 Skill 一致** —— Skill 1 用海底捞 queue 数据看排队，Skill 3 用海底捞 POI 数据规划路线。同一个 `shop_id` 同一份参数，**两个 Skill 自然一致**
3. **demo 调优极快** —— 录视频时发现"巴奴看起来不够拥挤" → 改一个 `initial_queue` 数字重录，不动代码

#### 同样的模式适用于所有业务实体

| Skill | 实体 | 参数控制什么 |
|---|---|---|
| Skill 1 | 餐厅 | 排队起点 + 速率 + 高峰窗口 + 突发事件 |
| Skill 2 | 外卖店 + 券池 | 菜单价 + 券面值 + 时段限制 + 满减门槛 |
| Skill 3 | POI（餐厅 / 影院 / 景点） | 评分 + 人均 + 营业时间 + 票务库存 |
| 所有 Skill | 用户偏好 | 饮食忌口 + 预算 + 习惯时段 + 交通方式 |

**核心 takeaway**：**写一次状态机基类，靠不同 JSON 参数文件生成 N 个 entity 的多样行为**。代码层只有一份，数据层无限可扩。

## 五、完整数据流（最关键的一张图）

```
┌─────────────────────────────────────────────────────────┐
│  评委 / Ray 在浏览器开沙盒 UI                              │
│  点击 [前进 10 分钟] 按钮                                  │
└────────────────────────┬────────────────────────────────┘
                         │ HTTP POST
                         ▼
┌─────────────────────────────────────────────────────────┐
│  sandbox/server.py                                       │
│  收到 /clock/advance?seconds=600                         │
│  调 clock.advance(600)                                   │
└────────────────────────┬────────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────────┐
│  mocks/clock.py（全局单例）                              │
│  _virtual_now += 600 秒                                  │
└────────────────────────┬────────────────────────────────┘
                         │ 60 秒后 cron 触发
                         ▼
┌─────────────────────────────────────────────────────────┐
│  Skill 1 的轮询 script                                   │
│  调 mocks.restaurants.queue_at("shop-001", virtual_now())│
└────────────────────────┬────────────────────────────────┘
                         │ 函数计算
                         ▼
┌─────────────────────────────────────────────────────────┐
│  mocks/restaurants.py                                    │
│  state(t) = 三种状态机组合算出"现在 5 桌"                  │
└────────────────────────┬────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│  Skill 1 发现 5 桌（满足触发条件）                         │
│  调 openclaw message send → 推送飞书消息                  │
│  "海底捞还有 5 桌就到你了，建议出发"                       │
└─────────────────────────────────────────────────────────┘
```

### 这条链路里的 4 个关键性质

1. **无数据库** —— Mock 不存数据，每次纯计算
2. **无副作用** —— 拨时间不改任何"业务事实"，只改"现在几点"。所以拨完再拨回去结果完全一样
3. **无并发问题** —— 所有 Skill 看到同一个 t，得到同一个结果
4. **完全可复现** —— 给定相同的 t，永远得到相同的状态

这 4 个性质共同决定了 Mock 状态机是**为 demo 而生的**：可预测、可控、可复现。

## 六、设计一份新 Mock 数据的具体步骤

写新 Mock 时按这个顺序思考——**这是 Ray 写 Skill 1 Mock 餐厅 / Lilian 写 Skill 2 Mock 券池时直接照搬的 checklist**。

### Step 1：先列"业务事件"（像写剧本）

什么时间应该发生什么？把它当成写电影剧本。

**例：Skill 1 海底捞场景**
- 16:00 餐厅开始营业，排队为 0
- 17:00 高峰开始，5 分钟内冲到 25 桌
- 17:00-19:00 持续高峰，队伍缓慢前进
- 18:45 注入"前面有人放弃"事件 → 跳号 -3
- 19:00-20:00 高峰退去，队伍加速前进
- 21:00 餐厅关门

### Step 2：把事件映射到三种状态机类型

| 业务现象 | 哪种状态机 |
|---|---|
| 排队基础递减 | 单调推进型 |
| 高峰期 rate 变化 | 时段周期型 |
| 18:45 跳号 | 事件触发型 |
| 21:00 关门后归零 | 时段周期型（按 hour cutoff） |

### Step 3：参数化（不要硬编码）

把规则写成 JSON 参数，不要硬编码到代码里：

```json
{
  "shop-001": {
    "name": "海底捞·王府井店",
    "open_hours": "16:00-21:00",
    "queue_machine": {
      "type": "monotonic_decay",
      "start_at": "17:00",
      "initial": 25,
      "base_rate_per_minute": 0.5,
      "peak_hours_modifier": {
        "17-19": 0.3,
        "default": 1.0
      }
    },
    "events": [
      {"time": "18:45", "type": "queue_jump", "delta": -3, "reason": "前面有人放弃"}
    ]
  }
}
```

**为什么参数化**：评委如果说"试试如果初始排队 50 桌会怎样" → 你可以**现场改参数文件**，不用动代码。这就是"可调试性"加分项。

### Step 4：预埋 demo 剧本关键瞬间

把剧本时间轴写到 `mocks/scenario.md`（人类可读）：

```markdown
# Demo 剧本时间轴 · 主线"用户外出吃饭"

| 虚拟时间 | 关键瞬间 | 触发什么 |
|---|---|---|
| 17:00 | 用户在飞书说"今晚海底捞" | Skill 1 启动监控（cron 注册） |
| 18:20 | 海底捞剩 5 桌 | Skill 1 自动取号 |
| 18:25 | 管家推送"建议出发" | 联动 Skill 1 + 高德路径 |
| 18:45 | (预埋事件) 前面有人放弃 → 跳 3 桌 | Skill 1 再次推送"提前 5 分钟到位" |
| 19:00 | 用户到店 | demo 故事弧结束 |
```

**好处**：
- 评委拨时间到 18:25 → 看到推送
- 评委拨到 18:45 → 看到跳号
- 评委拨到任何时间都能解释"为什么是这样"
- 团队队友 / 未来的 AI 看到这份剧本就懂"我们想演什么"

## 七、反模式 —— 这些坑必须避开

| 反模式 | 为什么是坑 | 应该怎么做 |
|---|---|---|
| 各 Skill 各搞各的时钟 | **联动崩盘**：Skill A 看到 18:30，Skill B 看到 18:35，两个 Skill 推的消息时间不一致 | 用唯一的 `mocks.clock.virtual_now()` |
| 在 Mock 里塞**业务逻辑**（如"如果用户是 VIP 给打折"） | Mock = 模拟**数据源**，不是模拟业务规则；业务规则在 Skill 里 | Mock 只返回 raw 状态；业务决策在 Skill 处理 |
| 用 `datetime.now()` | 评委拨时间无效 | 用 `virtual_now()` |
| 用 `random.random()` | demo 不可复现 | 用确定函数 |
| 让 LLM 生成 Mock 状态 | 同一时间两次得到不同结果 | 用预定义函数 |
| 一堆 Mock 数据**没注释 / 没剧本** | 队友 / 评委读不懂"为什么是这个值" | 在 `mocks/scenario.md` 写剧本时间轴 |
| 没有"测试拨时间"环节 | demo 当天才发现拨时间结果不对 | 每天 standup 前花 5 分钟拨一遍剧本验证 |
| 把所有事件堆在 demo 那 3 分钟 | 评委看不过来 / 显得 fake | 主线事件 1-2 个，其他作为"背景音"，详见剧本时间轴节奏设计 |

## 八、Mock 状态机的另一半：沙盒 UI

写完 Mock 状态机只是工程的一半。**另一半是沙盒 UI**——评委没有 Python 解释器，他们需要一个**视觉界面**才能拨时间。

### 沙盒 UI 的核心职责

| 职责 | 对应什么 |
|---|---|
| **看见当前虚拟时间** | 显示 `virtual_now()` 的值（大字号） |
| **拨虚拟时间** | 按钮调 `clock.advance(seconds)` |
| **看见各 Skill 当前状态** | 实时查各 mock 函数的当前值 |
| **看见待发推送队列** | 查 `openclaw cron list` |

Mock 状态机和沙盒 UI 是**互为前提**的：
- 没有 Mock 状态机 → 沙盒 UI 拨时间什么都不变（没意义）
- 没有沙盒 UI → Mock 状态机评委摸不到（白做）

**两者加起来才是"完整可演示的 Mock 系统"**。

详见 [`docs/design/sprint-plan.md`](design/sprint-plan.md) 的 "沙盒 UI MVP 具体长什么样" 一节。

## 九、（加分）状态机 Mock 与真后端的关系

虽然 hackathon 期间我们 Mock，但如果未来想接真美团 API，**架构应该长得几乎一样**：

```python
# 现在（Mock 阶段）
def get_queue(shop_id):
    return mocks.restaurants.queue_at(shop_id, virtual_now())

# 未来接真 API（仅替换实现，签名不变）
def get_queue(shop_id):
    return meituan_api.get_queue(shop_id)  # 真 API 不需要 t 参数
```

调用方（Skill 代码）**完全不变**。

这就是 Mock 状态机的隐藏价值：**它不只是过渡期的临时方案，它是"业务接口契约的占位实现"**——写好 Mock 等于写好了未来真接入时的接口规范。

## 十、最常见的 5 个问题（团队成员实际问过的）

### Q1：我能不能在 Mock 里加随机性，让数据看起来更"自然"？

**不要**。即使加一点点随机性也会让 demo 不可复现，录视频要拍 50 次。如果想让数据看起来不"太规律"，**用更复杂的事件触发型剧本**（多设计几个 events），不要用 random。

### Q2：Mock 里要不要存"用户做过什么"？比如用户取过号、点过外卖。

**不要**。Mock = 模拟外部数据源（餐厅、券池、地图等），用户行为是 OpenClaw memory + session 在管。Mock 不存任何用户产生的事实。

### Q3：3 个 Skill 的 Mock 数据要不要共享？比如 Skill 1 监控的餐厅在 Skill 3 行程规划里也出现？

**要共享**。同一家餐厅在 mocks/restaurants.json 里只定义一次，多个 Skill 都引用同一个 shop_id。这样跨 Skill 联动剧情才一致。

### Q4：评委如果说"试试如果初始排队 50 桌会怎样"，我怎么响应？

**改 mocks/restaurants.json 里的 `initial` 参数**，刷新沙盒 UI。这就是参数化设计的价值。

### Q5：mock 数据写得太"剧本化"会不会显得 fake？

**不会**，反而加分。冯岩评分项含"沙盒情况模拟是否详细"。一份详细的 `mocks/scenario.md` 剧本时间轴 = 直接展示"我们想清楚了 demo 要演什么"，是评委理解我们设计深度的关键载体。
