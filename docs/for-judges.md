# 给评委的项目说明 · 虾蜜 · 本地生活全天候私人管家

> 命题 01（基于 OpenClaw 的本地生活管家）。本文目标两个：**30 秒看懂这是什么**，以及**让你能自己跑起来看一个真实例子**。
> 配套：产品定义全文见 [`docs/design/skill-plan.md`](design/skill-plan.md)；三 Skill 能力速览见 [`docs/design/skills-overview-for-cases.md`](design/skills-overview-for-cases.md)。

## 0 · 一句话

**「虾蜜」是住在聊天软件里的本地生活管家**——它有自己的人设和记忆，把本地生活服务从"人找服务、自己串联"翻转成 **"服务主动找人"**。

三个本地生活 Skill，是同一个管家在你一天里不同场景主动出现：

| | Skill | 一句话 |
|---|---|---|
| 🔔 | **多店并行排队管家** | 后台同时盯多家餐厅排队，到点提醒出发，必要时（确认后）自动取号 |
| 🍱 | **智能外卖与采购助手** | 饭点/补货时少问少筛，直接给可确认的一键下单方案 |
| 🗺️ | **智能路线规划与分享** | 从查一家店到成局出行全覆盖；出行后还写总结、攒足迹、出年度报告 |

## 1 · 命题理解 & 我们的回答

冯岩对题面的本质判断（原话）：

> "本质上是我们现在的本地生活服务还停留在用户主动检索的阶段，**缺这样一个串联的过程**。"

我们的回答：三个 Skill **都有"主动出击"入口**，不只是被动应答——

- 观察到你在美团订座 → 静默启动该店排队监控
- 临近你惯常的饭点 / 常点店出了限时券 → 主动问要不要点
- 群里聊"周末去哪玩" → 自动整理成行程
- 行程结束 → 主动写一段能发朋友圈的小结

这就是"服务找人"：在你开口之前，管家已经在替你盯着。

## 2 · 怎么把它跑起来看

虾蜜是一个 **OpenClaw Agent**，运行逻辑全在后端、对话即 UI。提供两种方式，建议先跑 2a（最快、零依赖）。

### 2a · 沙盒控制台 —— 本地一条命令，零依赖，看清每个决策（建议先跑）

排队/券/票务这类数据**会随时间变化**（这正是命题的核心："排队接口在不同时间返回不同结果"）。沙盒让你**亲手拨时间验证**这一点，且**不需要部署、不需要 OpenClaw、不需要任何 API key**——只要本地有这份代码 + Python 3.10+。

**沙盒是什么**：`sandbox/` 目录两个文件——`server.py`（纯 Python 标准库 `http.server` 后端）+ `index.html`（前端面板）。它的关键设计是**不重写任何业务逻辑，而是直接调用三个 Skill 现成的脚本（subprocess 取 JSON）**，所以**面板里看到的 == 虾蜜实际拿到的**。

**启动**（在仓库任意位置都能跑，路径相对脚本自身解析）：

```bash
python3 sandbox/server.py          # 默认端口 8765，可加 --port 改
# 浏览器打开 http://127.0.0.1:8765
```

打开后你可以：

- **拨虚拟时钟** → 三个 Skill 面板的排队/券状态实时跟着变（亲手验证"同接口不同时间不同结果"）；
- **一键预设场景** → 跳到内置的几个 demo 关键瞬间；
- **因果可见** → 面板显示"因为改了 X → 管家把 Y 改成 Z"，决策过程透明。

> 沙盒直接跑三个 Skill 的脚本、不经过 OpenClaw，也不调高德（排队/券/票务都是本地 Mock），所以**离线就能完整复现**，拨两次同一时间得到同一结果。

### 2b · 部署到你自己的消息渠道 —— 完整的"对话"体验

虾蜜要"住进聊天软件"才能像管家一样对话——它**需要被部署到一个消息渠道（IM）里**才能交互（不是网页/CLI 直接聊）。你可以把它装进**你自己的 OpenClaw + 你自己的渠道**（前提：已安装 OpenClaw 并配好一个 LLM 后端 key，见 OpenClaw onboarding；这一步不像 2a 那样零依赖）：

1. 把三个 Skill 装进 OpenClaw（自包含 bundle）：
   ```bash
   python3 deploy.py watch-restaurant-queues meal-grocery-assistant route-planning-sharing
   ```
2. 把人设与记忆放进 OpenClaw 工作区：`openclaw-workspace/`（`SOUL.md` 人设 + `USER.md`/`MEMORY.md` 用户画像）→ `~/.openclaw/workspace/`，OpenClaw 每次会话自动注入；
3. 接入你自己的消息渠道：`openclaw channels add --channel <你的渠道>`；
4. 在该渠道里对虾蜜说话即可。

> 完整部署步骤见 [`docs/design/skill-deployment.md`](design/skill-deployment.md)。

## 3 · 三个 Skill 的核心设计逻辑

> 共同底座：统一人设「虾蜜」+ 统一记忆（`USER.md`/`MEMORY.md`，三 Skill 共读）+ 统一虚拟时钟。下面每个 Skill 都按 **定位 → 怎么触发 → 核心动作 → 设计亮点** 来读。

### 🔔 Skill 1 · 多店并行排队管家（`watch-restaurant-queues`）

- **定位**：后台同时盯多家候选餐厅的实时排队，到点提醒出发，必要时（两步确认后）自动取号。
- **怎么触发**：4 条**同级**入口——A 用户说"今晚吃 X 或 Y"；B（服务找人）观察到在美团订座 → 静默启动；C（服务找人）临近惯常饭点 → 主动问"要不要先盯排队"；D（服务找人）你提过的店今晚正好出窗口 → 主动推送。
- **核心动作**：搜店确认 → 并行监控多店（**未达阈值时静默不打扰**）→ 达阈值两步确认后取号 → 持续追踪前方桌数 → 剩 N 桌主动推送出发提醒（含高德真实步行/打车路径与费用）→ 联动叫车。
- **设计亮点**：监控按**虚拟时间推进**评估，能回答"**几点会跌到 N 桌**"——例如盯海底捞·望京店，18:00 排 32 桌，预测到约 18:55 跌到阈值就提醒你出发。这是"该出门没→快了→放出来了"的完整弧线，而不是一次性查询。

### 🍱 Skill 2 · 智能外卖与采购助手（`meal-grocery-assistant`）

- **定位**：饭点/补货场景的主动决策助手——少问、少筛，直接给**可确认的一键下单方案**。
- **怎么触发**：被动（"帮我点个饭""随便给个不踩雷的""想吃火锅""猫粮是不是该补了"）；主动（饭点 + 常点店有券 / 天气 / 日程显示明天有同品类局 / 采购周期到了）。
- **核心动作**：**三类外卖场景自动分流**——极度繁忙（1 个最稳一键下单）、纠结灵感（4 张推荐卡）、明确品类（常用 + 探索）；外加**采购补货**（周期感知，1-3 条建议）。避坑逻辑：昨日去重、天气避坑、日程避重。
- **设计亮点**：背后是一套**结构化用券引擎**——券的叠加/凑单/跨店比价/**时间杠杆**全部用整数分精确计算、确定性可复现（28 个单测全过）。能算出"等到 12:00 整点神券再下单省 ¥3"或"18:20 神券要秒空，尽快下"；还能主动扫临期券给"用掉它的最优凑单"——把"算不清能不能叠加"的累活替你干了。

### 🗺️ Skill 3 · 智能路线规划与分享（`route-planning-sharing`）

- **定位**：从"查一家店"到"成局出行"全覆盖的出行助手；出行后由虾蜜用闺蜜口吻写总结、并把足迹沉淀成档案与年度报告。
- **怎么触发**：用户主动问（"周末去哪玩""这家怎么去"）/ 群聊（被 @ 全力介入，命中关键词只发一条软提示）/ 日历到点但没安排路线。
- **核心动作**：按粒度裁剪——**轻**（查一家店/算一段路，一句话秒回）、**中**（按品类推 1 强推 + 1-2 备选，⭐主力）、**重**（多节点出行算 3-4 套路线 + 时间可行性校验）。分享阶段：发群版 + 朋友圈版文案 + 足迹年度报告。
- **设计亮点**：重粒度会**真算时间可行性**——"这么走赶不上 21:30 那场电影，帮你挑了 22:10 的"。所有距离/时长/评分/人均/打车费**来自高德真 API，可现场用高德 APP 比对验证**。

### 三条最强联动（"同一个管家串联服务"）

1. **Skill 3 规划出餐厅 → Skill 1 自动启动该店排队监控 + 出发提醒**（不用重复说）
2. **Skill 1 等位 > 30 分钟 + 饭点 → Skill 2 主动提示"先点份外卖垫一口"**
3. **Skill 3 行程结束 → Skill 2 主动问"明天补今天没买到的东西？"**

## 4 · 真数据 vs Mock（为什么这样 Mock）

我们**没有脱敏数据**，所以按命题要求自己 Mock——但**只 Mock 该 Mock 的**：

| 类型 | 内容 | 说明 |
|---|---|---|
| **真数据** | 距离 / 时长 / 评分 / 人均 / 营业时间 / 打车费（**高德 Web API**）；天气（**wttr.in**） | 凡是"可被你现场验证"的数字都走真 API——掏出高德 APP 能对上 |
| **Mock（随虚拟时钟）** | 排队桌数 / 优惠券 / 票务库存 | 这些是"会随时间变化"的运营层数据 |

**关键**：Mock 不是静态 JSON，也不是实时让大模型瞎编——而是**确定性状态机**（单调递减 / 时段周期 / 事件触发三类），由共享虚拟时钟驱动。所以**你把时间拨两次、得到的是同一个可复现的结果**，不会穿帮。

## 5 · 现场可复制的 Demo Prompts

> 用法：在沙盒（§2a）里把虚拟时间拨到「建议时间」（或点对应一键预设），然后对虾蜜说下面的话。

| # | 对虾蜜说 | 建议虚拟时间 | 预期 |
|---|---|---|---|
| 1 🔔 | "今晚和朋友去海底捞（望京），帮我盯着排队，快到了叫我" | 6/7 18:00 | 盯海底捞·望京店，32 桌递减；约 18:55 跌到阈值 → 主动提醒出发（含高德路径） |
| 2 🍱 | "中午太忙了，随便给我点个不踩雷的" | 6/7 12:00 | 极忙场景：直接给 1 个最稳的一键下单方案 |
| 3 🍱 | "想吃猪脚饭，哪家最划算" | 6/7 11:55 | 跨店券后比价 + 时间杠杆："等到 12:00 整点神券再下，省 ¥3" |
| 4 🍱 | "今晚有没有快过期的券，别浪费了" | 6/7 21:30 | 服务找人：扫到临期券 → 给"用掉它的最优凑单"主动推送 |
| 5 🗺️ | "周末想和闺蜜先吃火锅再看电影，帮我安排怎么走" | 周六 | 多节点行程：3-4 套路线 + 时间可行性（赶不上场次会改路线），数字来自高德 |
| 6 🔗 | （承接 #5）"就按这个安排，餐厅帮我盯着" | — | 联动：Skill 3 规划 → 自动启动 Skill 1 排队监控 |

> 评委可临场拨虚拟时间、临场换店名——demo 设计为可复现、扛得住。
